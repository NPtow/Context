# Test plan: `Context` memory system

## Scope
Проверяем не код, а работоспособность memory-архитектуры:
- навигацию;
- update protocol;
- retrieval protocol;
- temporal behavior;
- качество seed-контента.

## Test levels

### 1. Structure validation
Проверить, что структура репозитория соответствует плану и не смешивает уровни контекста.

Checks:
- корень содержит индексные файлы;
- есть отдельные слои для founder-memory, ventures, domains, sessions, archive;
- активный контекст отделен от архивного.

Commands:
- `cd /tmp/Context && find . -maxdepth 3 | sort`
- `cd /tmp/Context && rg -n "^# |^## " context-map.md 00_system 01_founder 02_ventures 03_domains 06_decisions 07_sessions`

Acceptance:
- структура читаема;
- путь к актуальному проекту очевиден;
- архив не выглядит как активный контекст.

### 2. Retrieval validation
Проверить, что типовые вопросы можно обслужить, прочитав ограниченное число файлов.

Scenarios:
- `что мы решили по рефералке`
- `какая сейчас главная гипотеза проекта`
- `что нового я понял про Никиту`
- `что мы уже знаем про Jack & Jill`
- `что изменилось с прошлой сессии`

Checks:
- для каждого вопроса есть явный путь чтения;
- нет необходимости читать весь session log;
- current truth отделен от raw evidence.

Acceptance:
- для типового вопроса достаточно 2-6 файлов;
- retrieval path описан в `00_system/retrieval-guide.md`.

### 3. Update validation
Проверить, что команда `обновись` интерпретируется однозначно.

Checks:
- описано, что обновляется всегда;
- описано, что обновляется только при новых устойчивых сигналах;
- описано, что делать, если активный проект неочевиден;
- описано, как session log поднимается в canonical.

Acceptance:
- другая LLM может выполнить `обновись` без длинных дополнительных указаний;
- нет конфликта между обновлением проекта и founder-memory.

### 4. Temporal validation
Проверить, что система хранит динамику и умеет работать с прошлым.

Checks:
- есть session layer;
- есть decision log;
- есть archive policy;
- есть способ ссылаться на прошлые проекты и прошлые решения.

Acceptance:
- можно ответить на вопрос `что мы думали раньше и что изменилось`;
- старые проекты не мешают чтению активного контекста.

## Negative cases
- Пользователь говорит `обновись`, но в диалоге два проекта с равной вероятностью.
  - Expected: один короткий уточняющий вопрос.
- В разговоре есть много идей, но мало устойчивых сигналов о Никите.
  - Expected: founder-memory не обновляется, только session log.
- Проект умер, но уроки по домену все еще ценны.
  - Expected: проект архивируется, а доменные выводы остаются в `03_domains/`.

## Release / execution gates
Перед первым push в `Context` должны выполняться все условия:
- есть `context-map.md`;
- есть `commands.md` и `retrieval-guide.md`;
- есть founder-memory seed;
- есть active project seed;
- есть session log seed;
- есть decision log;
- retrieval drills описаны в документации.

## Demo readiness checks
- Показать tree.
- Показать карту чтения для команды `что мы решили по рефералке`.
- Показать карту обновления для команды `обновись`.
- Показать, где лежит память о Никите и почему она не обновляется шумом.

## Non-goals of validation
- Не валидируем качество LLM как модели.
- Не валидируем embeddings / semantic search.
- Не валидируем автоматическую синхронизацию с внешними источниками.
